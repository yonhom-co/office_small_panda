## 二、整体架构设计（融合 Claude Code 式 Agent 架构）

### 2.0 设计哲学：从"固定 DAG"到"开放式 Agent Harness"

原计划以 LangGraph 预先编排的 DAG 状态机为主干——步骤在执行前就固化。但办公小浣熊面对的是**模糊目标、多变的真实数据、需要中途调整方向**的任务，预先固化的流程反而僵化。

借鉴 Claude Code 的核心思想，把主干换成 **Agent Harness（智能体外壳）+ tool-use 循环**，并用 **PocketFlow** 作为这套 Harness 的编排骨架（选型论证见 §2.4）：

> **核心循环**：读取用户输入 → 组装上下文 → LLM 思考并选择工具 → 执行工具并观测结果 → 把结果回灌上下文 → 重复，直到模型判定任务完成或请求人类决策。
>
> **PocketFlow 表达**：一个 `AgentNode` 的 `post()` 返回 `"continue"` 时路由回自身（自环），返回 `"done"` 时退出——tool-use 循环即"单节点自环"，是 PocketFlow 图原语最自然的用法。

这不是"先规划全部步骤再执行"，而是"边走边看、动态决策、随时可被人类打断修正"——这正是「DeepResearch 流程可交互 / 可调整 / 可追溯」的本质。规划器本身只是一个**工具**（`plan` 工具），而非全局控制器；人机决策通过 **Plan Mode** 实现；重活通过 **子代理（Sub-agent）** 隔离委派；上下文通过 **上下文工程** 控制成本与质量。

### 2.1 七大 Agent 架构要素 → 小浣熊映射

| Claude Code 架构要素 | 本质 | 在小浣熊中的落地 |
|----------------------|------|------------------|
| **① Agent Harness（tool-use 循环）** | LLM + 工具调用循环，动态决策而非固定流程 | 整个产品的运行内核；对话/创作空间/数据分析都跑在这套循环上 |
| **② 工具注册表（Tools）** | 能力以工具形式暴露，模型按需调用 | `upload_data` `run_code` `query_kb` `web_search` `make_chart` `write_doc` `gen_ppt` `plan` `todo` 等原子工具 |
| **③ 子代理委派（Sub-agents / Task）** | 把独立子任务交给隔离上下文的子代理，只回传结论，省主上下文 | 数据分析子代理、深度研究子代理、文档撰写子代理各自独立运转 |
| **④ Plan Mode（计划模式）** | 先出方案待人类批准，批准后才执行；计划可被修改 | 复刻「人机共决策」：复杂任务先展示可编辑计划，确认后再执行 |
| **⑤ 上下文工程（Context Engineering）** | 系统提示组装、记忆文件、超长对话压缩/摘要、分层上下文 | CLAUDE.md 式项目记忆、会话压缩、知识库按需注入而非全量塞入 |
| **⑥ Skills（按需加载能力包）** | 领域能力以 Markdown 指令包形式存在，触发时才载入上下文 | 五大行业场景模板（教育/医疗/电商/采购/财务）做成 Skills，按 `@` 触发 |
| **⑦ Hooks（确定性自动化）** | 把"每次 X 就 Y"的固定逻辑从模型剥离到外壳，可靠可审计 | 报告生成后自动归档、敏感数据自动脱敏、超时自动中止、操作审计 |
| **扩展：MCP（模型上下文协议）** | 标准协议接入外部数据源/工具 | 企业 ERP/CRM/OA 数据连接器、内部知识库接入，统一为 MCP server |
| **扩展：沙箱 + 权限** | 工具执行受权限与隔离约束 | 代码沙箱（数据分析）、文件读写权限、工具白名单、多租户隔离 |
| **扩展：Todo 自管理** | Agent 自维护任务清单，跟踪进度 | 复刻「任务规划/可追溯」：每步进度可视、可回看、可分支重跑 |

### 2.3 关键架构决策

| 决策点 | 选型 | 理由 |

### 2.4 选型论证：为什么用 PocketFlow 而非 LangGraph

> 基于对 PocketFlow 官方 README 与其 100 行核心源码（`pocketflow/__init__.py`）的核实。

**PocketFlow 是什么**：一个 100 行、零依赖、零厂商绑定的最小 LLM 编排框架。全部抽象仅 4 个原语——`Node`（`prep→exec→post→返回 action`，内置重试）、`Flow`（按 `action→successor` 路由的图执行器，`while curr` 循环驱动）、`BatchNode/AsyncNode/AsyncParallel*`（批处理/异步/并行）、`shared` 字典（节点间共享状态）。官方对比：LangGraph 37K 行/+51MB，PocketFlow 100 行/+56KB。

**采用 PocketFlow 的五点原因：**

1. **Claude Code 的内核本就是"手写循环"，不是图框架。** harness 是 `读上下文→LLM思考→选工具→执行→观测→回灌→循环`，本质是一个单节点自环：`AgentNode.post` 返回 `"continue"` 回自身、`"done"` 退出。PocketFlow 的 action 路由原生支持自环；LangGraph 反而要预设 StateGraph + 条件边 + reducer，把简单循环过度形式化。**用 PocketFlow 是"用最小图原语表达循环"；用 LangGraph 是"把循环塞进它预设的状态机模型"，经常打架。**

2. **"规划是工具，不是上帝"——PocketFlow 不强加流程模型。** §2.0 的核心论点是 `plan` 只是工具之一。PocketFlow 对流程组织零意见，harness 逻辑全在 `exec/post` 里。LangGraph 的 checkpointer/reducer/channel 语义会隐式约束状态结构，与"边走边看、动态 Replan"相抵触。

3. **零依赖、100 行、可审计——私有化与一体机的天然优势。** 办公小浣熊要交付私有化与软硬一体机。PocketFlow 56KB、无供应链风险、全代码可读，审计与内网离线打包极轻。LangGraph 拉 51MB 及一串依赖进一体机镜像是不必要的负担。**这与"每项目用 uv、不弄脏环境"的偏好一致。**

4. **大部分能力本就要自建，框架省的有限。** Hooks、MCP、沙箱、上下文压缩、Skills、Todo 自管理与编排框架无关，选谁都得自己写。LangGraph 真正多给的只有 **checkpointing / time-travel / 状态 schema 校验**；而"可追溯/分支重跑"对你只是把 `shared` 快照序列化进 Postgres——几十行。

5. **子代理隔离 = Flow-as-Node，原生支持。** PocketFlow 的 `Flow` 本身是 `BaseNode`，可嵌套。子代理即带独立 `shared` 副本的子 Flow，主 harness 通过节点调用、只取结论回传——直接对应 Claude Code 的 Task/子代理；并行子代理用 `AsyncParallelBatchNode`。

**Claude Code 模式 → PocketFlow 原语映射：**

| Claude Code 模式 | PocketFlow 实现 |
|---|---|
| tool-use 循环 | `AgentNode` 自环：`post` 返回 `"continue"`→自身，`"done"`→退出 |
| Plan Mode（人机共决策） | `PlanNode` 返回 `"await_approval"` 路由到 `HumanNode`；批准→`"exec"`，修改→`"replan"` |
| HITL 中断/恢复 | action 路由 + `shared` 快照持久化，恢复时从快照重建 Flow |
| 子代理委派 | 嵌套 `Flow` 作节点，隔离 `shared` 副本，只回传结论 |
| 并行子代理 | `AsyncParallelBatchNode` / `AsyncParallelBatchFlow` |
| 流式 + 可中断 | `AsyncNode.exec_async` 流式产出（官方有 streaming+interrupt cookbook） |
| Todo 自管理 | todos 存 `shared`，`TodoNode` 维护 |
| 可追溯/分支重跑 | **自建**：`shared` 快照序列化进 Postgres，按步回看/重放 |
| Hooks | **自建**：覆写 `Flow._orch` 或在节点 `post` 钩入事件总线 |
| MCP / 沙箱 / Skills | **与框架无关**，在 `exec` 里调用工具 |

**诚实付出的代价（PocketFlow 不给，需自建）：**

1. 无内置 checkpointing/time-trift → 自己序列化 `shared` 进 Postgres（几十行，可控）。
2. 无状态 schema/reducer 校验 → 自己管 `shared` 写入纪律（harness 完全自有，反而清晰）。
3. 无持久化执行器 → 长任务跨进程恢复基于快照重建（Plan Mode 恢复正需要，顺带做）。
4. 生态小 → 但走 OpenAI 兼容 + MCP，本就不依赖 LangChain 集成生态。
5. 并发/持久化边界场景少现成护栏 → 需自己写测试覆盖。

**净判断**：这些代价在 Claude Code 式"自研 harness"路线上几乎都是必经之路——选 LangGraph 也躲不掉 Hooks/MCP/沙箱/压缩的自建，只是多换回一个 checkpointer。**净收益为正，采用 PocketFlow。**

**接入代码骨架（Anthropic SDK + 原生 tool use）：**
```python
import os
from anthropic import Anthropic
from dotenv import load_dotenv

load_dotenv()

client = Anthropic(
    base_url=os.getenv("ARK_BASE_URL", "https://ark.cn-beijing.volces.com/api/coding"),
    auth_token=os.getenv("ARK_API_KEY"),   # Bearer 鉴权，用 auth_token 即可
)

MODEL = os.getenv("ARK_MODEL", "glm-5.2")
MAX_TOKENS = int(os.getenv("ARK_MAX_TOKENS", "8192"))

# tool-use 循环：模型返回 tool_use 块 → 执行工具 → 把 tool_result 回灌 → 继续
resp = client.messages.create(
    model=MODEL,
    max_tokens=MAX_TOKENS,
    tools=tool_schemas,          # Anthropic 原生 tool schema
    messages=msgs,
)
# resp.stop_reason == "tool_use" 时，从 resp.content 取 tool_use 块执行
```

**与 PocketFlow harness 的衔接**：`AgentNode.exec` 内调用上述 `messages.create`；`post` 依 `resp.stop_reason` 决定 action——`tool_use`→执行工具并回灌后返回 `"continue"`（自环），`end_turn`→`"done"`。原生 tool use 让 harness 不必自解析 function call，循环更干净。

**抽象层设计**：封装 `LLMProvider` 接口（默认 `ArkAnthropicProvider`）；私有化场景实现 `VLLMProvider`（本地 vLLM 的 OpenAI 兼容端点）或 `LocalAnthropicProvider`，通过环境变量切换，harness 与模型解耦。

**实测注意事项：**

1. coding plan 网关模型白名单会随时间变化，启动时调用 `GET /v1/models` 校验所选模型可用性，失败则回退。
2. 部分模型返回 `thinking` 块——harness 应保留并可用于「可追溯」展示，但回灌上下文时按需裁剪以控成本。
3. 部分模型输出可能含特殊 token（如 `</s>`、`</think>`），需在 harness 侧清洗。
4. **Embedding 实测受限**：coding plan 网关的 embedding 模型多为 `Retiring`；实测 ARK `/api/v3/embeddings` 下 `doubao-embedding-large-text-240915` 等均 `NotFound`，`doubao-embedding-vision-250615` 为 `ModelNotOpen`（模型存在但本账号未开通）。即**当前账号无可用 embedding 模型**。对策：知识库向量化（阶段 3）改用**本地 bge-m3**（CPU 可跑、中文好、私有化友好），不依赖 ARK embedding；`.env` 的 `EMBED_*` 改指向本地 sentence-transformers 服务。chat 走 ARK Anthropic、embedding 走本地，分而治之。

### 阶段 2：Plan Mode 人机共决策 + 子代理 + 上下文工程（2 周，P0）

**目标**：复刻功能①「深度思考 + 任务规划 + 方案输出」与优势②「DeepResearch 流程可交互」，让过程**可交互 / 可调整 / 可追溯**。这是 Claude Code 思想落地最关键的一阶段。

**步骤**：

2. **DeepResearch 子代理**：独立的"深度研究"子代理，自主拆解任务、多步检索与分析，关键节点回主循环请求人类决策；子代理隔离上下文，只回传结论。
3. **子代理调度器**：统一管理子代理生命周期、并发、上下文隔离与结果聚合。
4. **上下文工程**：系统提示分层组装；长对话自动**压缩/摘要**；知识库/记忆**按需注入**而非全量塞入；项目级记忆文件（类 CLAUDE.md）持久化用户偏好与项目背景。
5. **过程追溯**：每一步的输入、产出、工具调用、模型调用、代码均落库，支持回看、分支重跑、Replan。
6. **交付物生成器**：端到端产出**完整报告 / HTML 页面 / PPT**（python-pptx 生成 PPT），由主 Harness 在计划末段调度。

**步骤**：

1. **网页版**：对话式工作台、数据上传区、图表渲染区、任务计划面板（可交互编辑）、报告预览与导出。
2. **流式输出**：SSE 流式呈现思考过程与中间结果，体现"过程可追溯"。
3. **桌面端**：Tauri 封装网页版，增加本地文件访问。
4. **小程序版**：精简为问答 + 报告查看（受限于平台能力）。
5. **创作空间**：单空间内串联思路策划 → 提纲 → 内容 → 方案输出。

**验收标准**：网页版完整跑通端到端流程；桌面端可打包安装。

---

### 阶段 5：企业级能力 — MCP 扩展 + Hooks + 私有化 + 一体机（2.5 周）

**目标**：复刻"多种使用方式"中的企业部署形态，并补齐 Claude Code 式的 **MCP 扩展**与 **Hooks 确定性自动化**。

**步骤**：

1. **MCP 连接器**：把企业 ERP / CRM / OA / 内部知识库封装为 MCP Server，Agent 通过标准协议接入外部数据，无需改内核。
2. **Hooks 引擎**：事件总线 + 规则引擎，把"报告生成后自动归档""敏感数据自动脱敏""超时自动中止""每次工具调用写审计"等固定逻辑从模型剥离到外壳，可靠可审计。
3. **私有化部署**：全栈 Docker Compose / Helm Chart，离线镜像包，模型本地化（开源大模型 + 本地 Embedding）。
4. **模型抽象层**：`LLMProvider` 接口，默认 ARK Anthropic 端点；私有化切本地 vLLM（OpenAI 兼容）或本地 Anthropic 兼容服务。
5. **多租户与权限**：组织、团队、角色、数据隔离、工具白名单、审计日志。
6. **安全合规**：数据加密、沙箱强化（microVM）、敏感数据脱敏、操作审计。
7. **软硬一体机**：定义一体机硬件规格（GPU 型号、显存、存储），预装镜像，开机即用。

**验收标准**：在隔离内网环境一键部署成功，无外网依赖；MCP 连接器可拉取一份企业内部数据并分析；Hooks 规则可靠触发。

### 阶段 6：优化与行业场景打磨（持续）

**目标**：逼近官网精度指标，落地五大行业场景，建立 Agent 质量保障。

**步骤**：

1. **行业 Skills 填充**：教育 / 医疗 / 电商 / 采购 / 财务 五套 Skill 包（触发词 + 分析指令 + 报告样式 + 推荐工具序列）。
2. **精度提升**：针对时序计算、数据匹配、数理计算、异常检测做专项 Prompt + 工具链优化，目标接近宣称精度。
3. **Agent 评测集**：建立端到端评测基准——工具调用准确率、计划合理度、报告可用率、人工修改率、子代理委派是否得当。
4. **上下文成本治理**：监控 Token 消耗、压缩命中率、子代理回传精简度，防止上下文膨胀。
5. **性能优化**：大文件流式处理、图表懒渲染、沙箱预热池、并发。
6. **可观测性**：调用链追踪、Token 成本统计、质量监控、Hooks 触发审计。

## 六、风险与对策

| 风险 | 对策 |
|------|------|
| LLM 生成的数据分析代码不可靠 | 沙箱执行 + 结果校验 + 失败自愈重试；关键步骤 Plan Mode 人机确认 |
| 代码沙箱安全（注入/逃逸） | microVM 隔离、禁网、资源限额、只读挂载、白名单库 |
| **Agent 失控/死循环/工具滥用** | 循环步数与 Token 上限、工具权限白名单、Plan Mode 卡关键决策、Hooks 超时硬中止 |
| **上下文膨胀/成本失控** | 子代理隔离上下文只回传结论、长对话自动压缩、按需注入、Token 成本监控 |
| **子代理委派不当（该自己做却委派/反之）** | 委派策略 Prompt 约束 + 评测集回归 |
| 私有化模型能力不足 | 模型抽象层可切换；提供"在线强模型 / 离线开源模型"双档 |
| 大文件/大数据性能 | 流式解析、采样探查、polars 加速、超阈值分块 |
| 报告/PPT 还原度 | 模板化 + 占位符注入，保证版式稳定 |
| 多端维护成本 | 前端核心逻辑复用，桌面/小程序薄封装 |

## 八、Claude Code 架构思想融合要点（速查）

> 本节是把现代 Agent 思想落到本项目的"一句话清单"，供开发时对照。

1. **Harness 是地基**：先用 **PocketFlow** 把 tool-use 循环跑通，再谈功能；功能 = 工具 + 子代理 + Skills 的组合，而非硬编码流程。
2. **规划是工具，不是上帝**：`plan` 只是 Agent 可调用的工具之一；真正的"流程控制"是循环本身 + Plan Mode 的人类闸门。
3. **Plan Mode = 人机共决策**：复杂任务先暂停出方案，人改完再执行——直接对应办公小浣熊「DeepResearch 可交互/可调整」。
4. **子代理保护主上下文**：数据分析、深度研究、文案都委派给隔离子代理，只回传结论，避免上下文被原始数据淹没。
5. **上下文工程决定质量与成本**：分层提示 + 长对话压缩 + 按需注入 + 项目记忆文件。
6. **Skills 让行业扩展零侵入**：教育/医疗/电商/采购/财务 做成按需加载包，`@` 触发，不占基础上下文。
7. **Hooks 把确定性还给外壳**：归档、脱敏、审计、超时等"每次必做"的逻辑不交给模型，用 Hooks 可靠执行。
8. **MCP 统一外部接入**：企业数据源标准化为 MCP Server，内核不改即可扩展。
9. **沙箱 + 权限是底线**：Agent 能调用工具，但每个工具都受权限与隔离约束。
10. **Todo 自管理 = 可追溯**：Agent 自维护任务清单，每步可回看、可分支重跑。
